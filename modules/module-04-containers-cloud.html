Containers, Kubernetes & the cloud around your binary.

Every listing says "cloud-native." Go's single static binary makes it the easiest language to containerize well. Here's Docker, Kubernetes, CI/CD, and AWS — from a Go engineer's angle.

Concept 1

The multi-stage Dockerfile

Go compiles to one static binary — so your final image can be almost empty. Build in a Go image, copy just the binary into a tiny base.

# stage 1: build
FROM golang:1.25 AS build
WORKDIR /src
COPY go.mod go.sum ./
RUN go mod download          # cached layer
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

# stage 2: run — tiny & secure
FROM gcr.io/distroless/static
COPY --from=build /app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]
Why it matters
CGO_ENABLED=0 makes a fully static binary so it runs on distroless/static or scratch — a ~10 MB image with no shell, no OS packages, tiny attack surface. Copying go.mod first caches dependency downloads across builds.

Concept 2

Config from the environment

Containers are immutable; configuration comes from the environment (the 12-factor rule). Read it at startup, fail fast if required values are missing.

port := os.Getenv("PORT")
if port == "" { port = "8080" }          // sensible default
dsn := os.Getenv("DATABASE_URL")
if dsn == "" { log.Fatal("DATABASE_URL required") }  // fail fast
Gotcha
Never bake secrets into the image. In Kubernetes, inject them via Secret objects mounted as env vars; in AWS, use Secrets Manager / SSM Parameter Store. A committed DATABASE_URL with a password is the classic security-review finding.

Concept 3

Kubernetes in one breath

You rarely write raw k8s from scratch as a backend dev, but you must read it. The unit is a Pod (your container); a Deployment keeps N replicas running; a Service gives them a stable address.

apiVersion: apps/v1
kind: Deployment
spec:
  replicas: 3
  template:
    spec:
      containers:
        - name: weather
          image: registry/weather:v1.4.0
          ports: [{ containerPort: 8080 }]

Concept 4 · the interview favorite

Probes & resource limits

Kubernetes needs to know if your app is alive and ready. This is where your /healthz endpoint from L03 earns its keep.

          livenessProbe:                # restart if this fails
            httpGet: { path: /healthz, port: 8080 }
          readinessProbe:               # remove from load balancer if not ready
            httpGet: { path: /readyz,  port: 8080 }
          resources:
            requests: { cpu: "100m", memory: "64Mi" }
            limits:   { cpu: "500m", memory: "256Mi" }
Liveness vs readiness
Liveness = "am I deadlocked? restart me." Readiness = "can I serve traffic right now?" — return not-ready while warming up or if the DB is down, and k8s stops routing to you without killing you. Confusing the two causes restart loops. Expect this question.
Gotcha
Memory limits matter for Go: exceed the limit and the container is OOM-killed. Set GOMEMLIMIT (Go 1.19+) near the container limit so the garbage collector works harder before the kernel kills you.

Concept 5

CI/CD: the pipeline

Every commit should test, vet, build, and (on main) ship. A minimal GitHub Actions pipeline for Go:

- run: go vet ./...
- run: go test -race -cover ./...
- run: go build -o app ./cmd/server
- run: docker build -t registry/weather:${{ github.sha }} .
- run: docker push registry/weather:${{ github.sha }}
Idiom
Run tests with -race in CI — the detector catches concurrency bugs (L02) that pass locally. Tag images with the git SHA for traceable, rollback-able deploys.

Concept 6

The cloud, mostly AWS

UK and NL listings lean AWS hardest. As a Go dev you'll meet: ECS/EKS (run containers), RDS (managed Postgres), S3 (blobs), SQS/SNS (queues), CloudWatch (logs/metrics). The AWS SDK for Go v2 is the client.

cfg, _ := config.LoadDefaultConfig(ctx)   // creds from env/role
s3c := s3.NewFromConfig(cfg)
_, err := s3c.PutObject(ctx, &s3.PutObjectInput{...})
Idiom
In production, don't use access keys — attach an IAM role to the task/pod and the SDK picks up temporary credentials automatically. "Least-privilege IAM" is a phrase worth understanding for interviews.

Evidence-based practice

Retrieval drills

Predict before revealing. Drill 1 recalls L03.

Explain · recalls L03

Your readiness probe hits /readyz. What should that handler actually check — and how does it differ from /healthz?

/healthz (liveness) should be cheap and almost always return 200 — it only says "the process isn't wedged." /readyz (readiness) should verify dependencies you need to serve: can you ping the DB? is the cache reachable? Return 503 from /readyz when a dependency is down, and k8s pulls you from the load balancer without restarting you.
Find the flaw

Why is this Dockerfile a bad idea for a Go service?

FROM golang:1.25
COPY . .
RUN go build -o app .
CMD ["./app"]
It ships the entire Go toolchain (~800 MB) in the final image — huge, slow to pull, and a big attack surface (compilers, shell, package manager all present). Use a multi-stage build: compile in golang, then copy only the binary into distroless/static or scratch for a ~10 MB image running as non-root.
Explain the trade-off

Why set GOMEMLIMIT instead of just trusting Go's GC in a container?

Go's GC targets a heap-growth ratio, not an absolute ceiling — it doesn't know about the container's memory limit. Under load it can let the heap grow past the k8s limit, and the kernel OOM-kills the pod (a hard crash, dropped requests). GOMEMLIMIT tells the GC a soft ceiling so it collects more aggressively as you approach it — trading some CPU for staying alive. Set it a bit below the container limit.
Verify
Build your image, then docker images to see its size — under ~20 MB means the multi-stage build worked. Run it locally with docker run -e PORT=8080 -p 8080:8080 weather and hit /healthz.
