A production-shaped URL shortener.

A blazing redirect hot path with Redis, links in Postgres, click events on Kafka, JWT on the management API — and a bonus: this is the classic system-design interview question, so you'll whiteboard the thing you actually built. Applies L03 · APIs, L05 · Data, and L07 · Security.

🔨 Build module 📦 Applies L03 · L05 · L07 ⏱ ~1 week hands-on

What you'll ship

  • POST /v1/links — create a short link (JWT-protected)
  • GET /{code} — the public redirect hot path, Redis-cached
  • A gRPC service exposing link management to internal callers
  • Postgres persistence of links, via a repository
  • Click events on Kafka — analytics without slowing redirects
  • /healthz + /readyz, graceful shutdown, layered architecture
handler ──▶ service ──▶ cache (Redis, hot codes) │ │ ├─▶ repo ──▶ Postgres (links) │ │ └─▶ Kafka (click events, async) └─ HTTP + gRPC both call the same service
01

Lay out the packages

Structure by responsibility, not by type. This layering is what makes it testable (L06) and is the shape reviewers expect.

cmd/server/main.go     // wiring: config, deps, start
internal/shortener/
   service.go          // business logic, depends on interfaces
   http.go             // REST handlers + redirect (L03)
   grpc.go             // gRPC server (L03)
   repo.go             // Postgres impl of LinkStore (L05)
   cache.go            // Redis impl of Cache (L05)
   events.go           // Kafka click publisher (L05)
Grill
Defend the boundary: why does service depend on a LinkStore interface rather than importing the Postgres repo directly? (Answer from L06: so tests inject a fake and so you could swap Postgres for something else — dependency inversion.)
02

The redirect hot path (cache-aside)

Redirects outnumber link creations a thousand to one, so this path gets the L05 cache-aside treatment: check Redis → on miss read Postgres and populate. Latency here is the product.

func (s *Service) Resolve(ctx context.Context, code string) (string, error) {
    if url, ok := s.cache.Get(ctx, code); ok {
        return url, nil                     // cache hit — no DB touched
    }
    url, err := s.store.Lookup(ctx, code)  // miss → Postgres
    if err != nil {
        return "", err                      // includes ErrNotFound
    }
    s.cache.Set(ctx, code, url, time.Hour)
    return url, nil
}

// the handler: resolve, then redirect
http.Redirect(w, r, url, http.StatusFound)   // 302 — on purpose
Grill
Why 302 Found and not 301 Moved Permanently? (Browsers cache a 301 forever and skip your server on the next click — you lose every future click event and the ability to retarget the link. The status code is a product decision.)
03

Creating links: random codes, collision-safe

Generate short codes from crypto/rand, base62-encoded. A unique constraint in Postgres is your safety net: on the (rare) collision, generate again and retry.

const alphabet = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

func newCode(n int) (string, error) {
    b := make([]byte, n)
    for i := range b {
        idx, err := rand.Int(rand.Reader, big.NewInt(62))
        if err != nil { return "", err }
        b[i] = alphabet[idx.Int64()]
    }
    return string(b), nil          // 62^7 ≈ 3.5 trillion codes
}

// insert; on unique-violation, regenerate and retry (bounded times)
Grill
Two to defend: (1) Why random codes instead of an auto-incrementing ID encoded in base62? (Sequential codes let anyone enumerate every link ever created — a real leak; L07 thinking applied to product design.) (2) Why is the DB's unique constraint, not an "exists?" check before insert, the correct collision defense? (Check-then-insert has a race window; the constraint is atomic.)
04

Expose it over REST and gRPC

Thin transports: parse, call the service, translate domain errors into each protocol's vocabulary (L03). Validate the submitted URL — you're about to redirect strangers to it.

mux.HandleFunc("POST /v1/links", func(w http.ResponseWriter, r *http.Request) {
    // decode {"url": "..."}; url.Parse + require http/https scheme (L07: validate input)
    link, err := svc.Create(r.Context(), body.URL)
    switch {
    case errors.Is(err, ErrInvalidURL):
        http.Error(w, "invalid url", http.StatusBadRequest)
    case err != nil:
        http.Error(w, "internal error", http.StatusInternalServerError)
    default:
        w.WriteHeader(http.StatusCreated)
        json.NewEncoder(w).Encode(link)   // {"code":"x7Kf9Qa", ...}
    }
})

// gRPC: same service, different error vocabulary
func (g *grpcServer) Create(ctx context.Context, in *pb.CreateRequest) (*pb.Link, error) {
    link, err := g.svc.Create(ctx, in.Url)
    if errors.Is(err, ErrInvalidURL) {
        return nil, status.Error(codes.InvalidArgument, "invalid url")
    }
    return toProto(link), err
}
Grill
Why map ErrInvalidURL to 400 in REST but codes.InvalidArgument in gRPC? (Each protocol has its own error vocabulary — the service returns domain errors; each transport translates. Leaking SQL errors to clients is the anti-pattern.) Bonus L07 grill: should you accept a javascript: URL? Why is scheme validation a security control here, not pedantry?
05

Click events — off the hot path

Analytics wants every click; the redirect wants none of that latency. Publish {code, ts, referrer} to Kafka asynchronously — the redirect never waits on the broker.

func (s *Service) recordClick(code, ref string) {
    select {
    case s.clicks <- Click{code, time.Now(), ref}:  // buffered channel
    default:                                        // full? drop, don't block
        s.dropped.Add(1)                            // ...but count the drops
    }
}
// a background goroutine (L02!) drains s.clicks into Kafka in batches
Grill
Defend the default branch: you're deliberately dropping data under load. Why is that correct here — and why would the same choice be a firing offense in the fintech route's ledger? (Click analytics tolerates loss; money doesn't. Knowing which guarantees each domain needs is the actual skill.)
Verify
docker compose up Postgres+Redis (from L04), create a link, then hit GET /{code} twice — confirm the second resolve is a cache HIT in your logs and the browser lands on the target. Stop Redis and confirm redirects still work (slower) from Postgres; stop Postgres and confirm /readyz flips to 503.
06

Protect the management API

Apply L07: JWT middleware in front of POST /v1/links and the gRPC management calls. The redirect stays public — that's the product — and /healthz + /readyz stay open for the probes.

api := authn(apiMux)             // verify JWT, attach caller to ctx
mux.Handle("/v1/", api)          // management: token required
mux.HandleFunc("GET /{code}", redirect)   // public hot path
srv.Handler = logging(recover(mux))
// on SIGTERM: srv.Shutdown(ctx) to finish in-flight requests
Grill
This service has two publics: an open redirect path and a locked management path. What's the abuse case for each — and which one needs rate limiting more? (The open redirect invites spammers creating malicious links via stolen tokens and bots hammering resolution; per-route thinking beats one global policy.)
Verify
Three curls: POST /v1/links with no token → 401; tampered token → 401; valid token → 201. Then confirm GET /{code} works with no token, and /readyz returns 200 unauthenticated — or your P04 deploy will never become ready.
