Concept 1 · the framework

How to run a design interview

Interviewers grade your process, not a memorized answer. A reliable order:

1. Clarify requirements (functional + non-functional) → 2. Estimate scale (QPS, data size) → 3. Sketch the high-level components → 4. Drill into data model & APIs → 5. Identify bottlenecks & scale them → 6. Discuss trade-offs.

Idiom
Always separate functional requirements ("return current weather") from non-functional ones ("p99 < 100ms, 10k QPS, 99.9% uptime"). The non-functional numbers drive every architecture decision that follows.

Concept 2

Vertical vs horizontal scaling

Vertical = bigger machine (simple, but a ceiling and a single point of failure). Horizontal = more machines (the cloud-native default from L04). Go's cheap concurrency and small binaries make horizontal scaling especially easy.

Key enabler
Horizontal scaling requires stateless services — any instance can handle any request. That's why session state lives in Redis (L05) and auth prefers stateless JWTs (L07): so you can add replicas behind a load balancer without sticky sessions.

Concept 3

Load balancing

A load balancer spreads traffic across your stateless replicas and removes unhealthy ones — using the very /readyz probe you built in L04/P03.

            ┌─ weather-1
client ─▶ LB ─┼─ weather-2   // round-robin / least-conn
            └─ weather-3   // unhealthy → LB stops routing (readyz)
Idiom
In Kubernetes the Service + ingress is your load balancer. Common algorithms: round-robin (default), least-connections (better for uneven request cost), and consistent hashing (when you need a client to stick to a shard).

Concept 4

Caching strategy

Caching is the highest-leverage scaling move. You already built cache-aside (L05); at scale you also think about where to cache and how to invalidate.

Layers: client → CDN (for public GETs) → your Redis cache → database. Each layer it doesn't reach is load you didn't pay for.

Gotcha
"There are only two hard things in CS: cache invalidation and naming things." Stale data is the tax on caching. Decide per-endpoint: short TTLs (weather: 10 min is fine), event-based invalidation, or accept eventual consistency. Never cache anything user-specific in a shared/CDN layer.

Concept 5

Data at scale

When one Postgres can't keep up:

Read replicas — send reads to followers, writes to the primary (weather is read-heavy → big win). Sharding/partitioning — split data by key (e.g. by region) across databases when write volume or size exceeds one node. Indexing — the cheapest first fix; a missing index turns a fast query into a table scan.

Order of operations
Scale data in this order: add an index → add a cache (L05) → add read replicas → shard. Sharding is powerful but adds huge complexity (cross-shard queries, rebalancing) — reach for it last, and be able to say why in an interview.

Concept 6 · the theory they ask about

CAP & consistency

The CAP theorem: in a network partition you must choose Consistency or Availability — you can't have both while Partition-tolerant (and networks always partition eventually).

How to apply it
A weather read can be AP — serve slightly stale data, stay available. A bank transfer must be CP — refuse rather than double-spend. The senior move is saying "different parts of one system make different CAP choices": the weather cache is AP, the payment ledger is CP. That nuance is what fintech interviewers (PT/LU) listen for.

Retrieval drills

Explain · recalls L05/L07

Your weather service is at 3 replicas behind a load balancer, but users randomly get logged out. What's the likely cause?

Session state is stored in-memory per instance, so a request routed to a different replica than the login doesn't find the session. Fix by making the service stateless: move sessions to shared Redis (L05) or use stateless JWTs (L07). Sticky sessions "fix" it too but undermine even load distribution — statelessness is the right answer.
Estimate

10,000 weather requests/sec, 90% served from a 10-minute cache. Roughly how many requests/sec actually hit the upstream/DB?

~1,000/sec at the cache miss rate (10% of 10k) — but with a 10-minute TTL and a limited set of cities, real misses are far lower, since a popular city is fetched once per 10 min and served from cache to everyone else. This is the point of caching: a small backend handling a huge front-end load. (Watch for the thundering-herd on expiry — L05.)
Explain the trade-off

Why might you make the weather endpoint AP but the billing endpoint CP in the same system?

They have different correctness requirements. Weather can be slightly stale without harm, so favor availability — serve cached/replica data even during a partition. Billing must never double-charge or lose money, so favor consistency — refuse the operation rather than risk a wrong result. One system, two CAP choices, each matched to the cost of being wrong.
Verify
Take your P03/P04 service and write a one-page design doc: requirements, a component diagram, where each cache/replica goes, and the CAP choice per endpoint. Being able to draw and defend it is exactly the interview.
