Build a Weather CLI in Go.

A hands-on build. You'll write every line yourself; this module gives the plan, the grilling, and the checkpoints. It applies concept module L01 · Foundations.

🔨 Build module 📦 Applies L01 ⏱ ~60 min hands-on

What you'll ship

  • A command weather London that prints live conditions
  • A real HTTP call, JSON decoded into a struct
  • Honest error handling and a non-zero exit code on failure
  • A Go module you can go build into a single binary
01

Scaffold the module

Every Go project starts as a module. In your empty repo:

$ go mod init example/weather
$ touch main.go

That go.mod file is your project's identity and dependency ledger — the equivalent of package.json or requirements.txt, but built into the toolchain.

Verify
Run go run . now. Empty main.go won't compile yet — read the error. Getting comfortable with Go's compiler messages early pays off all module long.
02

Read the city from the CLI

Before writing this, predict: with go run . London, which index of os.Args holds "London"? (Recall from L01: [0] is the program itself.)

Grill
Design decision to defend before you code: what should happen when the user runs weather with no city? Print usage and exit non-zero? Default to a city? Decide now — that choice shapes your main.
func main() {
    if len(os.Args) < 2 {
        fmt.Fprintln(os.Stderr, "usage: weather <city>")
        os.Exit(1)   // non-zero = failure, the Unix way
    }
    city := os.Args[1]
    // ... next step wires the fetch
}
03

Call the weather API

Use a free endpoint like wttr.in (no key needed) or Open-Meteo. The key learning here is the error idiom and defer from L01 — every I/O call returns an error you must handle.

func fetch(city string) ([]byte, error) {
    url := fmt.Sprintf("https://wttr.in/%s?format=j1", city)
    resp, err := http.Get(url)
    if err != nil {
        return nil, fmt.Errorf("requesting weather: %w", err)
    }
    defer resp.Body.Close()   // runs on return — no leaks
    if resp.StatusCode != 200 {
        return nil, fmt.Errorf("weather API returned %d", resp.StatusCode)
    }
    return io.ReadAll(resp.Body)
}
Grill
Why return ([]byte, error) instead of printing inside fetch? (Answer: separation of concerns — fetch gets data, main decides how to present and how to fail. This is what makes it testable in Project P3.)
04

Decode JSON into a struct

This is the L01 payoff: struct tags. The API returns nested JSON; you model only the fields you need and tag them to match.

type Report struct {
    Current []struct {
        TempC string `json:"temp_C"`
        Desc  []struct{ Value string `json:"value"` } `json:"weatherDesc"`
    } `json:"current_condition"`
}

var r Report
if err := json.Unmarshal(data, &r); err != nil {
    return fmt.Errorf("decoding response: %w", err)
}
Verify
go run . London should now print a real temperature. If you get a zero value where data should be, your struct tag doesn't match the JSON key — inspect the raw response with go run . London | head temporarily.
05

Present & ship

Format the output, exit non-zero on any error, then build a real binary.

$ go build -o weather .
$ ./weather Lisbon
Lisbon: 24°C, Sunny
$ echo $?      # 0 on success, 1 on failure
Ship checklist
gofmt -w . (canonical formatting) · go vet ./... (catches bugs) · try an invalid city and confirm you get a clean error + non-zero exit, not a panic.
