Concept 1 · deepens L01

Error wrapping done properly

L01 introduced if err != nil. Idiomatic Go goes further: wrap errors with context using %w, then let callers inspect them with errors.Is and errors.As instead of string matching.

var ErrNotFound = errors.New("not found")   // a sentinel

func load(id string) (Report, error) {
    r, err := db.Get(id)
    if err != nil {
        return Report{}, fmt.Errorf("load %s: %w", id, err)  // wrap
    }
    ...
}
// caller — inspect the wrapped chain:
if errors.Is(err, ErrNotFound) { /* handle 404 */ }
var pgErr *pq.Error
if errors.As(err, &pgErr) { /* got the concrete DB error */ }
Gotcha
%w wraps (preserves the chain for errors.Is/As); %v flattens to a string and loses it. Use %w at most once per Errorf. And never compare errors with err.Error() == "not found" — string matching breaks the moment a message changes.

Concept 2

Interface design: small & consumer-side

The Go proverb: "The bigger the interface, the weaker the abstraction." Define interfaces where they're used (the consumer), not where the type is implemented, and keep them tiny — often one method.

// Good: defined by the consumer, minimal
type Storer interface {
    Save(ctx context.Context, r Report) error
}
// "Accept interfaces, return structs"
func NewService(s Storer) *Service { return &Service{store: s} }
Idiom
Don't pre-emptively define an interface for every struct — that's a Java habit. Add one only when you have a second implementation or you need to mock for tests (L06). Premature interfaces are an over-engineering smell reviewers flag.

Concept 3 · Go 1.18+

Generics, used sparingly

Generics let you write type-safe reusable code without interface{} + casts. They shine for data structures and collection helpers, not everywhere.

// a generic Map over a slice — one impl, any type
func Map[T, U any](in []T, f func(T) U) []U {
    out := make([]U, len(in))
    for i, v := range in { out[i] = f(v) }
    return out
}
names := Map(reports, func(r Report) string { return r.City })
Gotcha
Don't reach for generics reflexively. If an interface expresses the behavior clearly, prefer it. The Go team's guidance: use generics when you'd otherwise copy-paste a function for many types, or for general-purpose containers — not to make ordinary business logic "flexible."

Concept 4 · the classic interview trap

Slice internals

A slice is a header — pointer, length, capacity — over a backing array. Understanding this explains Go's most surprising behaviors.

a := []int{1, 2, 3, 4}
b := a[:2]              // shares the SAME backing array
b = append(b, 99)      // cap allows it → overwrites a[2]!
// a is now [1 2 99 4]
Gotcha
Because slices share backing arrays, appending to a sub-slice can mutate the original. When you need an independent copy, use slices.Clone or append([]T(nil), src...). And append may or may not reallocate depending on capacity — always assign the result back: s = append(s, x).

Concept 5

Maps, nil, and ordering

Reading a missing key returns the zero value (no panic); the comma-ok form tells you if it existed. But writing to a nil map panics.

var m map[string]int     // nil map
n := m["x"]              // OK: returns 0
m["x"] = 1             // PANIC: assignment to nil map
m = make(map[string]int) // fix: make it first
Gotcha
Map iteration order is randomized by design — never rely on it. To output sorted, collect keys into a slice and slices.Sort them. Also: maps aren't safe for concurrent write (L02) — guard with a sync.Mutex or use sync.Map.

Concept 6

Project layout & modules

Reviewers judge structure at a glance. The conventions:

cmd/<app>/main.go   // entrypoints, thin
internal/...        // private packages — unimportable by others
pkg/... (optional)  // deliberately public libraries
go.mod / go.sum     // module + checksum lockfile
Idiom
Put code under internal/ to make it un-importable outside your module — a compiler-enforced privacy boundary. Name packages for what they provide (weather), not layers (utils, helpers, models are anti-patterns). Manage deps with go get / go mod tidy; commit go.sum.

Evidence-based practice

Retrieval drills

Predict before revealing. Drill 1 recalls L01/L06.

Predict the output · recalls slices

What is a after this runs?

a := []int{1, 2, 3, 4, 5}
b := a[:3]
b = append(b, 100)
fmt.Println(a)
output → [1 2 3 100 5]. b has len 3 but cap 5 (shares a's array), so append writes into the existing backing array at index 3 — overwriting a[3] (which was 4) with 100. No reallocation happened because capacity allowed it.
Find the bug · recalls L06

Why is this error check unreliable?

if err != nil && err.Error() == "not found" {
    return http.StatusNotFound
}
It matches the error's string, which breaks if the message is reworded or wrapped with context ("load 42: not found" won't equal "not found"). Use a sentinel and errors.Is(err, ErrNotFound), which walks the wrapped chain regardless of surrounding text. String-matching errors is a guaranteed review comment.
Explain the trade-off

A teammate defines an interface with 8 methods for their one struct. What's the idiomatic critique?

A big interface with a single implementation adds indirection for no abstraction — "the bigger the interface, the weaker the abstraction." Idiomatic Go defines small interfaces at the consumer, listing only the methods that consumer needs, and only when there's a real second implementation or a test to mock. Prefer returning the concrete struct and letting callers declare their own minimal interface.
Verify
Run go vet ./... and gofmt -l . on your project — both should be silent. Then try the slice-aliasing trap in the Go playground and watch the original mutate. Seeing it beats reading it.
