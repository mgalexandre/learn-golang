Track B · Projects · P02 · intermediate

A concurrent multi-city fetcher.

Take P01's fetch and run it across dozens of cities at once — safely bounded, with timeouts and clean partial-failure handling. Applies concept module L02 · Concurrency.

🔨 Build module 📦 Applies L02 ⏱ ~90 min hands-on

What you'll ship

  • weather London Lisbon Paris ... fetched concurrently
  • A bounded worker pool — never more than N in flight
  • A per-request context timeout so one slow city can't stall the batch
  • A rate limiter to stay under the API's cap
  • Partial results: one city failing doesn't sink the rest
01

Make fetch context-aware

First refactor P01's fetch to take a context.Context as its first parameter (the L02/L03 convention). This is what lets the caller impose a timeout.

func fetch(ctx context.Context, city string) (Report, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET", urlFor(city), nil)
    resp, err := http.DefaultClient.Do(req)
    // ... same decode as P01
}
Grill
Predict before building: if you give each city a 2-second timeout but run them sequentially, how long does a 30-city batch take worst case? And concurrently with a pool of 5? (That gap is the whole point of this project.)
02

Build the worker pool

Recreate the L02 pattern: a jobs channel of cities, N workers, a results channel. Bounding concurrency is what keeps you from opening 500 sockets at once.

type result struct{ city string; report Report; err error }

jobs := make(chan string)
out  := make(chan result)

var wg sync.WaitGroup
for i := 0; i < 5; i++ {          // 5 workers
    wg.Add(1)
    go func() {
        defer wg.Done()
        for city := range jobs {
            rep, err := fetch(ctx, city)
            out <- result{city, rep, err}   // carry the error, don't crash
        }
    }()
}
Grill
Why send the error inside the result struct instead of logging it in the worker? (Answer: the collector decides how to report; a failed city becomes a row in the output, not a lost goroutine — that's "partial results.")
03

Feed jobs, close, collect

The classic choreography: a goroutine closes out once all workers finish, so the collector's range terminates cleanly.

go func() {                     // producer
    for _, c := range cities { jobs <- c }
    close(jobs)                    // no more work → workers exit their range
}()
go func() { wg.Wait(); close(out) }()  // all done → stop the collector

for r := range out {           // collect until out closes
    if r.err != nil {
        fmt.Printf("%-10s ERROR: %v\n", r.city, r.err)
        continue
    }
    fmt.Printf("%-10s %s\n", r.city, r.report.Summary())
}
Verify
Run with a bad city mixed in: go run . London NotACity Paris. You should see two good rows and one ERROR row — proof that one failure doesn't abort the batch. Then go run -race . to confirm no data races on the shared channels.
04

Add a rate limiter

Free APIs cap requests. A time.Ticker (or golang.org/x/time/rate) throttles workers so you fetch, say, 5 per second regardless of pool size.

limiter := time.NewTicker(200 * time.Millisecond)  // ~5 req/s
defer limiter.Stop()
// inside the worker, before fetch:
<-limiter.C   // block until the next tick
Grill
Trade-off to reason about: more workers make you faster, but the rate limiter caps throughput anyway. So what's the right worker count? (Hint: enough to keep the limiter saturated while requests are in-flight, not more.)
05

Consider errgroup

Once you understand the manual version, meet golang.org/x/sync/errgroup — it bundles the WaitGroup, error propagation, and context cancellation. If one task fails, it can cancel the rest.

g, ctx := errgroup.WithContext(ctx)
g.SetLimit(5)                     // built-in concurrency bound
for _, city := range cities {
    g.Go(func() error { return fetchInto(ctx, city, &results) })
}
err := g.Wait()   // first non-nil error, or nil
Ship checklist
go vet ./... · go test -race ./... · time it: time go run . <20 cities> should be roughly (slowest city), not the sum. That speedup is the concurrency win — the thing interviewers want you to explain.
