Concept 1

Goroutines: cheap, not free

A goroutine is a function scheduled by the Go runtime, not the OS. It starts at a few KB of stack and grows on demand, so you can run hundreds of thousands. You launch one with go:

go fetch("London")   // runs concurrently; main does NOT wait
fmt.Println("kicked off")
Gotcha
When main returns, the program exits — even if goroutines are still running. A bare go f() with nothing to wait on often prints nothing. You need a way to wait (a channel or sync.WaitGroup).

↳ Coming from JS: this is not async/await. There's no event loop and no promise — a goroutine is genuinely parallel across CPU cores.

Concept 2

Channels: how goroutines talk

Instead of sharing memory and locking it, Go's motto is "share memory by communicating." A channel is a typed pipe. Send with ch <- v, receive with v := <-ch.

results := make(chan string)   // unbuffered

go func() {
    results <- fetch("London")  // blocks until someone receives
}()

msg := <-results             // blocks until a value arrives
fmt.Println(msg)
Idiom
An unbuffered channel is a rendezvous: send and receive happen together, synchronizing the two goroutines. A buffered channel — make(chan string, 5) — only blocks the sender when the buffer is full. Use buffering to decouple producer and consumer speed.

Close a channel when no more values will be sent. Ranging over it then drains cleanly:

close(results)
for msg := range results {  // exits when channel is closed & empty
    fmt.Println(msg)
}
Gotcha
Sending on a closed channel panics. Closing a channel twice panics. The rule: only the sender closes, and only once. Receivers use the comma-ok form v, ok := <-ch to detect closure.

Concept 3

select: waiting on many things

select blocks until one of several channel operations is ready — the concurrency equivalent of switch. It's how you add timeouts and cancellation.

select {
case msg := <-results:
    fmt.Println("got", msg)
case <-time.After(2 * time.Second):
    fmt.Println("timed out")   // don't wait forever
default:
    fmt.Println("nothing ready")   // makes select non-blocking
}
Idiom
A select with a time.After case is the canonical timeout pattern. Add a default case and select stops blocking entirely — useful for "try to send, but drop it if nobody's listening."

Concept 4

sync, and the race you can't see

Sometimes you really do share memory — a counter, a cache. Two goroutines touching the same variable, at least one writing, is a data race: undefined behavior. Go ships a detector: go run -race .

var (
    mu    sync.Mutex
    count int
)
func incr() {
    mu.Lock()
    count++          // critical section — one goroutine at a time
    mu.Unlock()
}

To wait for a batch of goroutines, use sync.WaitGroup:

var wg sync.WaitGroup
for _, city := range cities {
    wg.Add(1)
    go func() {
        defer wg.Done()
        fetch(city)   // Go 1.22+: loop var is safe to capture
    }()
}
wg.Wait()   // blocks until every Done() is called
Gotcha
Before Go 1.22, capturing the loop variable city in the goroutine was the most infamous Go bug — every goroutine saw the last value. Go 1.22+ makes each iteration its own variable, so this is finally safe. Know this: interviewers still ask about it.

Concept 5 · the production one

context: cancellation & deadlines

Every real Go service threads a context.Context through its call chain. It carries cancellation, deadlines, and request-scoped values. When a request is cancelled (client hangs up, timeout hits), the context's Done() channel closes and every goroutine downstream can bail out.

ctx, cancel := context.WithTimeout(context.Background(), 3*time.Second)
defer cancel()   // always cancel to release resources

req, _ := http.NewRequestWithContext(ctx, "GET", url, nil)
resp, err := http.DefaultClient.Do(req)
if err != nil {
    // if ctx expired, err wraps context.DeadlineExceeded
    return err
}
Idiom
Convention: ctx context.Context is the first parameter of any function that does I/O or blocks. Never store a context in a struct. Always call the cancel func (a defer cancel() right after creation). This pattern shows up in nearly every job description as "context-aware services."

Putting it together

The worker pool

This is the pattern behind Project P2, and a favorite whiteboard question: bounded concurrency. Instead of one goroutine per task (unbounded), you run N workers pulling from a jobs channel.

jobs := make(chan string, len(cities))
out  := make(chan string, len(cities))

for w := 0; w < 4; w++ {          // 4 workers, not 10000
    go func() {
        for city := range jobs {       // pull until jobs closes
            out <- fetch(city)
        }
    }()
}
for _, c := range cities { jobs <- c }
close(jobs)                          // tells workers: no more work
for i := 0; i < len(cities); i++ { fmt.Println(<-out) }
Why it matters
Real APIs rate-limit you. A worker pool caps how many requests fly at once — the difference between "fast" and "banned." Bounded concurrency + context timeouts is exactly what "scalable, resilient services" means on a Go job spec.

Evidence-based practice

Retrieval drills

Predict the output · recalls L01

What does this print, and why is it a trap?

func main() {
    go fmt.Println("hello from goroutine")
    fmt.Println("hello from main")
}
output → usually just hello from main. main returns before the goroutine is scheduled, so the program exits and the goroutine never runs. It's a race with the exit — output is non-deterministic. Fix: a WaitGroup or a channel receive.
Find the bug

This deadlocks. What's wrong?

func main() {
    ch := make(chan int)
    ch <- 42          // send
    fmt.Println(<-ch)   // receive
}
panic → fatal error: all goroutines are asleep - deadlock!
An unbuffered channel needs a sender and receiver ready at the same time. Here main blocks on the send forever — there's no other goroutine to receive. Fix: send from a separate goroutine, or make it buffered (make(chan int, 1)).
Explain the trade-off

When would you reach for a Mutex instead of a channel?

Channels shine for transferring ownership of data or coordinating flow (pipelines, worker pools). A Mutex is simpler and faster when you're just guarding shared state in place — a counter, a map cache — and there's no handoff. Rule of thumb: "channels for communication, mutexes for state." Overusing channels for simple counters is a common over-engineering smell.
Verify
Prove a race exists before you trust the fix: write two goroutines incrementing a shared int without a lock, run go run -race ., read the report, then add the Mutex and watch it go green.
