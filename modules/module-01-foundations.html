Build 01 · concept

package, main, and go run

Every Go executable is a package named main with a function main(). There's no top-level script body like Python or Node — execution starts at main, full stop.

package main

import (
    "fmt"
    "os"
)

func main() {
    // os.Args[0] is the program path — real args start at [1]
    city := "the world"
    if len(os.Args) > 1 {
        city = os.Args[1]
    }
    fmt.Printf("weather for %s\n", city)
}
Gotcha
os.Args[0] is the program itself, not your first argument. So go run . London puts "London" at index 1. Reading os.Args[1] without the length check panics when no city is given.

Language core

Declarations & zero values

Go has no undefined. Every variable starts at its type's zero value: 0, "", false, or nil. This is why Go code rarely guards against "was this set?"

var count int        // 0
var name string      // "" — not nil
ready := false       // := infers type, functions only
Gotcha
An unused local variable or import is a compile error, not a warning. Coming from JS/Python this bites early — the compiler simply refuses to build.

↳ Coming from Python/JS: := is "declare & assign" (inside functions); = is "reassign". Mixing them up is the #1 beginner error.

The Go mindset

Errors are values, not exceptions

There is no try/catch. Functions that can fail return (result, error), and you handle the error right there. It's verbose on purpose: the failure path is always visible.

resp, err := http.Get(url)
if err != nil {
    return fmt.Errorf("fetching %s: %w", url, err)
}
defer resp.Body.Close()  // runs when the function returns
Idiom
if err != nil { return ... } is the heartbeat of Go. %w wraps the underlying error so callers can inspect it. defer schedules cleanup that runs on the way out — the Go answer to finally.

Build 03 · concept

Structs & struct tags

Go has no classes. You group data in a struct, and attach behavior with methods. Capitalized fields are exported (public); lowercase are package-private — visibility is spelled by the letter case.

type Weather struct {
    City string  `json:"name"`   // tag maps JSON key → field
    Temp float64 `json:"temp"`
}

w := Weather{City: "London", Temp: 12.5}

Those backtick struct tags are how encoding/json knows that the API's "name" field fills your City field. No decorators, no schema library — just metadata on the field.

Idiom
Value receiver (w Weather) works on a copy (read-only); pointer receiver (w *Weather) can mutate the original. Pick pointer when you change state or the struct is large — and stay consistent within a type.

Evidence-based practice

Retrieval drills

Don't re-read — retrieve. Predict each answer before revealing. Getting it wrong and correcting is what makes it stick.

Predict the output

Running go run . London, what prints?

func main() {
    fmt.Println(len(os.Args), os.Args[0] != "")
}
output → 2 true
os.Args holds the program path and "London" → length 2. os.Args[0] (the path) is non-empty, so true.
Find the bug

This won't compile. Why?

import "fmt"

func main() {
    city := "Paris"
    fmt.Println("hello")
}
city is declared but never used → compile error: declared and not used. Go treats unused locals (and imports) as build-breaking. Use it, or remove it.
Explain the trade-off

Why does Go return (value, error) instead of throwing exceptions?

The failure path becomes explicit and local — you can't accidentally ignore it the way an uncaught exception silently unwinds the stack. Trade-off: more verbose code (if err != nil everywhere) in exchange for control flow you can read top-to-bottom. Reserve panic for truly unrecoverable states.
Verify
Every concept ends at the terminal. In your repo: go mod init example/weather, then go run . London. Real compiler output is the best teacher — bring the errors back and we'll read them together.
