Concept 1

A server in the standard library

No framework required — net/http is production-grade. A handler is anything with ServeHTTP(w, r); HandlerFunc lets a plain function qualify.

func health(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
    fmt.Fprintln(w, "ok")
}

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("GET /healthz", health)  // method+path (Go 1.22+)
    srv := &http.Server{
        Addr:         ":8080",
        Handler:      mux,
        ReadTimeout:  5 * time.Second,
        WriteTimeout: 10 * time.Second,   // never trust the network
    }
    log.Fatal(srv.ListenAndServe())
}
Gotcha
The default http.ListenAndServe(":8080", mux) has no timeouts — a slow client can hold a connection forever (Slowloris). Always construct an http.Server with explicit ReadTimeout/WriteTimeout. Interviewers love this one.

Concept 2

Routing & JSON responses

Since Go 1.22 the standard ServeMux understands methods and path wildcards, so you rarely need a third-party router.

mux.HandleFunc("GET /weather/{city}", func(w http.ResponseWriter, r *http.Request) {
    city := r.PathValue("city")          // typed path param
    report, err := store.Get(r.Context(), city)  // pass the request ctx!
    if err != nil {
        http.Error(w, "not found", http.StatusNotFound)
        return
    }
    w.Header().Set("Content-Type", "application/json")
    json.NewEncoder(w).Encode(report)     // stream, don't buffer
})
Idiom
Thread r.Context() into every downstream call. When the client disconnects, that context is cancelled and your DB query / upstream call can abort — the L02 context lesson, now in its natural habitat.

Concept 3

Middleware is just a function

Cross-cutting concerns — logging, auth, recovery — wrap a handler and return a handler. No magic, no decorators.

func logging(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        start := time.Now()
        next.ServeHTTP(w, r)
        log.Printf("%s %s %v", r.Method, r.URL.Path, time.Since(start))
    })
}
// wrap: srv.Handler = logging(recover(mux))
Idiom
Middleware composes by nesting: logging(auth(mux)). Order matters — a recover middleware outermost catches panics from everything inside, turning a crash into a clean 500.

Concept 4

Designing the REST surface

Employers say "REST APIs at scale." What they mean: predictable resources, correct status codes, and versioning.

Status codes that mean something

200 ok · 201 created · 400 bad input · 401/403 auth · 404 missing · 409 conflict · 422 validation · 500 your fault · 503 dependency down. Returning 200 with an error body is an anti-pattern.

Idiom
Version at the path (/v1/weather) or a header. Return errors as a consistent JSON shape ({"error": {"code": "...", "message": "..."}}) so clients can handle them programmatically.

Concept 5

gRPC: contracts between services

For service-to-service traffic, gRPC (protobuf over HTTP/2) is faster and strongly typed. You define the contract once in a .proto file and generate Go code with protoc.

// weather.proto
service Weather {
  rpc Get(GetRequest) returns (Report);
}
message GetRequest { string city = 1; }
message Report     { double temp_c = 1; string desc = 2; }
When to use which
REST/JSON for public & browser-facing APIs (human-readable, cache-friendly). gRPC for internal microservice calls (typed, streaming, low overhead). Many Go shops expose REST at the edge and gRPC between services — you'll see both on one job spec.

...and GraphQL

A few listings (LU/BE) pair "REST, GraphQL." GraphQL exposes one endpoint where the client asks for exactly the fields it wants — solving over- and under-fetching for rich frontends. In Go, gqlgen generates type-safe resolvers from a schema.

# schema.graphql
type Query { weather(city: String!): Report }
type Report { tempC: Float!, description: String! }
Trade-off
GraphQL's flexibility costs you: HTTP caching is harder (everything's a POST to one URL), and a naive resolver hitting the DB per field causes the N+1 query problem — fixed with a dataloader that batches. Reach for GraphQL when clients need flexible, nested reads; stick with REST for simple resource CRUD.

Concept 6 · the distributed-systems part

Failure is the default

Across a network, every call can be slow, fail, or duplicate. "Resilient services" on a job spec means you handle these deliberately.

Timeouts & retries with backoff

ctx, cancel := context.WithTimeout(r.Context(), 2*time.Second)
defer cancel()
for attempt := 0; attempt < 3; attempt++ {
    err := callUpstream(ctx)
    if err == nil { break }
    time.Sleep(time.Duration(attempt*attempt) * 100 * time.Millisecond) // backoff
}
Gotcha
Only retry idempotent operations. Retrying a "charge card" that actually succeeded but timed out charges twice. The fix is an idempotency key so the server dedupes. This is core fintech knowledge — heavily represented in the PT/LU listings.

Beyond retries: circuit breakers (stop hammering a dead dependency), rate limiting (protect yourself), and graceful shutdown (srv.Shutdown(ctx) to drain in-flight requests on deploy).

Evidence-based practice

Retrieval drills

Predict before revealing. Drill 1 recalls L02.

Explain · recalls L02

A request handler starts 3 goroutines to call upstreams. The client disconnects. How do the goroutines know to stop?

They watch r.Context().Done(). When the client disconnects, the server cancels the request context; its Done() channel closes, and any goroutine passed that ctx (or a call using it, like an HTTP request built with NewRequestWithContext) unblocks and returns context.Canceled. Without threading the context through, they'd leak.
Find the bug

Why might this handler leak goroutines / connections under load?

mux.HandleFunc("GET /slow", func(w http.ResponseWriter, r *http.Request) {
    resp, _ := http.Get("https://upstream/data")
    io.Copy(w, resp.Body)
})
Three defects: (1) the error is ignored — resp could be nil → panic; (2) resp.Body is never Close()d → connection leak; (3) no context/timeout, so a slow upstream ties up this handler indefinitely. Fix: check the error, defer resp.Body.Close(), and use http.NewRequestWithContext(r.Context(), ...).
Explain the trade-off

Your service calls a flaky payment provider. Why add a circuit breaker on top of retries?

Retries help with transient blips, but if the provider is genuinely down, every request retrying 3× multiplies load on a dying dependency and ties up your own goroutines — a cascading failure. A circuit breaker "opens" after N failures and fails fast for a cooldown, giving the dependency room to recover and keeping your service responsive. Retries + breaker together, not either alone.
Verify
Run your server, then curl -i localhost:8080/weather/London. Check the status code and Content-Type. Then kill the upstream and confirm you get a clean 503, not a hang or a panic.
