Track B · Fintech route · P02 · intermediate

A concurrent portfolio pricer.

Take P01's fetch and revalue a whole portfolio of currency positions at once — safely bounded, with timeouts and clean partial-failure handling. Applies concept module L02 · Concurrency.

🔨 Build module 📦 Applies L02 ⏱ ~90 min hands-on

What you'll ship

  • pricer EURUSD GBPUSD CHFUSD ... quoted concurrently
  • A bounded worker pool — never more than N in flight
  • A per-request context timeout so one slow quote can't stall the batch
  • A rate limiter to stay under the API's cap
  • Partial results: an unpriced position is reported, never guessed
01

Make fetch context-aware

First refactor P01's fetch to take a context.Context as its first parameter (the L02/L03 convention). This is what lets the caller impose a timeout.

func fetch(ctx context.Context, pair string) (Quote, error) {
    req, _ := http.NewRequestWithContext(ctx, "GET", urlFor(pair), nil)
    resp, err := http.DefaultClient.Do(req)
    // ... same decode as P01
}
Grill
Predict before building: if you give each quote a 2-second timeout but run them sequentially, how long does a 30-position portfolio take worst case? And concurrently with a pool of 5? (That gap is the whole point of this project.)
02

Build the worker pool

Recreate the L02 pattern: a jobs channel of pairs, N workers, a results channel. Bounding concurrency is what keeps you from opening 500 sockets at once.

type result struct{ pair string; quote Quote; err error }

jobs := make(chan string)
out  := make(chan result)

var wg sync.WaitGroup
for i := 0; i < 5; i++ {          // 5 workers
    wg.Add(1)
    go func() {
        defer wg.Done()
        for pair := range jobs {
            q, err := fetch(ctx, pair)
            out <- result{pair, q, err}   // carry the error, don't crash
        }
    }()
}
Grill
Why send the error inside the result struct instead of logging it in the worker? In finance the answer has teeth: a position with no fresh price must show up as UNPRICED in the report — silently dropping it would misstate the portfolio's value.
03

Feed jobs, close, collect — and total

The classic choreography: a goroutine closes out once all workers finish, so the collector's range terminates cleanly. The collector also totals the priced positions.

go func() {                     // producer
    for _, p := range pairs { jobs <- p }
    close(jobs)                    // no more work → workers exit their range
}()
go func() { wg.Wait(); close(out) }()  // all done → stop the collector

var total float64
unpriced := 0
for r := range out {           // collect until out closes
    if r.err != nil {
        fmt.Printf("%-8s UNPRICED: %v\n", r.pair, r.err)
        unpriced++
        continue
    }
    total += r.quote.Value()
    fmt.Printf("%-8s %.4f\n", r.pair, r.quote.Rate())
}
fmt.Printf("total %.2f (%d positions unpriced)\n", total, unpriced)
Verify
Run with a bogus pair mixed in: go run . EURUSD XXXYYY GBPUSD. You should see two priced rows, one UNPRICED row, and an honest summary line. Then go run -race . to confirm no data races. (Notice total is only touched by the collector goroutine — that's why there's no race.)
04

Add a rate limiter

Free APIs cap requests. A time.Ticker (or golang.org/x/time/rate) throttles workers so you fetch, say, 5 per second regardless of pool size.

limiter := time.NewTicker(200 * time.Millisecond)  // ~5 req/s
defer limiter.Stop()
// inside the worker, before fetch:
<-limiter.C   // block until the next tick
Grill
Trade-off to reason about: more workers make you faster, but the rate limiter caps throughput anyway. So what's the right worker count? (Hint: enough to keep the limiter saturated while requests are in-flight, not more.)
05

All-or-nothing mode with errgroup

Once you understand the manual version, meet golang.org/x/sync/errgroup — it bundles the WaitGroup, error propagation, and context cancellation. If one task fails, it cancels the rest — which gives your pricer a second personality.

g, ctx := errgroup.WithContext(ctx)
g.SetLimit(5)                     // built-in concurrency bound
for _, pair := range pairs {
    g.Go(func() error { return priceInto(ctx, pair, &book) })
}
err := g.Wait()   // first non-nil error, or nil
Grill
You now have two behaviors: step 03's partial results and errgroup's fail-fast. When would a risk system prefer aborting the whole revaluation over a partial one? (When a partial total could be mistaken for the real total — sometimes no answer beats a wrong-looking answer.) Defend which mode your CLI should default to.
Ship checklist
go vet ./... · go test -race ./... · time it: time go run . <20 pairs> should be roughly (slowest quote), not the sum. That speedup is the concurrency win — the thing interviewers want you to explain.
