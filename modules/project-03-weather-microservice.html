Track B · Weather route · P03 · advanced

A production-shaped weather microservice.

Wrap your fetcher in an HTTP + gRPC API, cache in Redis, persist to Postgres, secure it, and structure it in clean layers. Applies L03 · APIs, L05 · Data, and L07 · Security.

🔨 Build module 📦 Applies L03 · L05 · L07 ⏱ ~1 week hands-on

What you'll ship

  • GET /v1/weather/{city} REST endpoint returning JSON
  • A gRPC service exposing the same data to internal callers
  • Redis cache-aside in front of an upstream API
  • Postgres persistence of every reading, via a repository
  • /healthz + /readyz, graceful shutdown, layered architecture
handler ──▶ service ──▶ cache (Redis) │ │ └─▶ upstream API (P02 fetcher) │ └─▶ repo ──▶ Postgres └─ HTTP + gRPC both call the same service
01

Lay out the packages

Structure by responsibility, not by type. This layering is what makes it testable (L06) and is the shape reviewers expect.

cmd/server/main.go     // wiring: config, deps, start
internal/weather/
   service.go          // business logic, depends on interfaces
   http.go             // REST handlers (L03)
   grpc.go             // gRPC server (L03)
   repo.go             // Postgres impl of ReadingStore (L05)
   cache.go            // Redis impl of Cache (L05)
Grill
Defend the boundary: why does service depend on a ReadingStore interface rather than importing the Postgres repo directly? (Answer from L06: so tests inject a fake and so you could swap Postgres for something else — dependency inversion.)
02

The service layer with cache-aside

The service orchestrates: check cache → on miss fetch + persist + populate cache. It knows nothing about HTTP or SQL — just interfaces.

func (s *Service) Get(ctx context.Context, city string) (Report, error) {
    if r, ok := s.cache.Get(ctx, city); ok {
        return r, nil                       // cache hit (L05)
    }
    r, err := s.upstream.Fetch(ctx, city)   // P02 fetcher, ctx-aware
    if err != nil {
        return Report{}, err
    }
    _ = s.store.Save(ctx, r)               // persist (best-effort)
    s.cache.Set(ctx, city, r, 10*time.Minute)
    return r, nil
}
03

Expose it over REST

Thin handlers: parse, call the service, map errors to status codes (L03). No business logic here.

mux.HandleFunc("GET /v1/weather/{city}", func(w http.ResponseWriter, r *http.Request) {
    rep, err := svc.Get(r.Context(), r.PathValue("city"))
    switch {
    case errors.Is(err, ErrNotFound):
        http.Error(w, "unknown city", http.StatusNotFound)
    case err != nil:
        http.Error(w, "upstream unavailable", http.StatusServiceUnavailable)
    default:
        w.Header().Set("Content-Type", "application/json")
        json.NewEncoder(w).Encode(rep)
    }
})
04

Add the gRPC surface

Define the .proto (L03), generate code, and implement the server — reusing the exact same svc.Get. Two protocols, one brain.

func (g *grpcServer) Get(ctx context.Context, in *pb.GetRequest) (*pb.Report, error) {
    rep, err := g.svc.Get(ctx, in.City)
    if errors.Is(err, ErrNotFound) {
        return nil, status.Error(codes.NotFound, "unknown city")
    }
    return toProto(rep), err
}
Grill
Why map ErrNotFound to http.StatusNotFound in REST but codes.NotFound in gRPC? (Because each protocol has its own error vocabulary — the service returns domain errors; each transport translates. Leaking SQL errors to clients is the anti-pattern.)
05

Health, readiness & graceful shutdown

Prepare for L04/P04: readiness reflects real dependencies; shutdown drains in-flight requests on SIGTERM.

mux.HandleFunc("GET /readyz", func(w http.ResponseWriter, r *http.Request) {
    if err := db.PingContext(r.Context()); err != nil {
        http.Error(w, "db down", http.StatusServiceUnavailable)  // pulled from LB
        return
    }
    w.WriteHeader(http.StatusOK)
})
// on SIGTERM: srv.Shutdown(ctx) to finish in-flight requests
Verify
docker compose up Postgres+Redis (from L04), run the service, then: hit /v1/weather/London twice and confirm the second is a cache HIT in your logs; stop Redis and confirm it still serves from upstream; stop Postgres and confirm /readyz flips to 503.
06

Protect it with auth

Apply L07: put an authentication middleware in front of write/private routes so only valid callers get through. Keep /healthz and /readyz open (the probes need them).

protected := authn(mux)          // verify JWT/session, attach user to ctx
srv.Handler = logging(recover(protected))

// authn: missing/invalid token → 401; valid → next with user in context
Grill
Design decision to defend: should /healthz sit inside or outside the auth middleware? (Outside — Kubernetes probes send no credentials, so an authenticated health check would fail readiness and your pod would never go live. L04 + L07 collide here.)
Verify
Three curls: a protected route with no token → 401; with a tampered token → 401; with a valid token → 200. And confirm /readyz still returns 200 with no token, or your P04 deploy will never become ready.
