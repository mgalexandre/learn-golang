Concept 1 · vocabulary

Authentication vs authorization

Two words interviewers expect you to distinguish precisely. Authentication (AuthN) = who are you? Authorization (AuthZ) = what are you allowed to do? A valid login (AuthN) doesn't grant admin rights (AuthZ).

In an HTTP service both live in middleware (L03): authenticate the caller, attach their identity to the request context, then let handlers check permissions.

func authn(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        user, err := verifyToken(r.Header().Get("Authorization"))
        if err != nil {
            http.Error(w, "unauthorized", http.StatusUnauthorized) // 401
            return
        }
        ctx := context.WithValue(r.Context(), userKey, user)
        next.ServeHTTP(w, r.WithContext(ctx))
    })
}
Idiom
401 Unauthorized means "I don't know who you are" (AuthN failed). 403 Forbidden means "I know you, but you can't do this" (AuthZ failed). Swapping them is a classic tell.

Concept 2

JWTs & sessions

A JWT is a signed token carrying claims (user id, expiry, roles). The server verifies the signature — no database lookup needed, which is why it scales for stateless microservices.

token, err := jwt.Parse(raw, func(t *jwt.Token) (any, error) {
    return publicKey, nil            // verify signature
})
if err != nil || !token.Valid {
    return nil, ErrUnauthorized
}
Gotcha
Three JWT footguns interviewers probe: (1) never accept the alg: none algorithm — pin the expected algorithm; (2) always check exp (expiry); (3) JWTs can't be revoked before expiry, so keep them short-lived and pair with a refresh token. A long-lived JWT is a liability.

The alternative — server-side sessions (a random ID in a cookie, state in Redis from L05) — is revocable instantly but needs a lookup per request. Stateless JWT vs stateful session is a real trade-off, not a default.

Concept 3

OAuth2 & OpenID Connect

You rarely build login from scratch — you delegate to an identity provider (Google, Auth0, Keycloak) via OAuth2 (authorization) and OIDC (an identity layer on top). Go's golang.org/x/oauth2 handles the dance.

The flow in one line
User → redirected to provider → logs in → provider redirects back with a code → your server exchanges the code for tokens (server-to-server, with your client secret). Know the term Authorization Code flow — it's the one to use for web apps; implicit flow is deprecated.

Concept 4 · never get this wrong

Password hashing

If you store passwords, hash them with a slow, salted algorithm built for the job — bcrypt, scrypt, or argon2. Never MD5/SHA-256 (too fast → brute-forceable), never plaintext, never your own crypto.

hash, _ := bcrypt.GenerateFromPassword([]byte(pw), bcrypt.DefaultCost)
// store `hash`. To check:
err := bcrypt.CompareHashAndPassword(hash, []byte(attempt))
if err != nil { /* wrong password */ }
Gotcha
bcrypt salts automatically — don't add your own. And compare secrets with bcrypt.CompareHashAndPassword or crypto/subtle.ConstantTimeCompare, never ==, to avoid timing attacks that leak how much of the secret matched.

Concept 5

TLS & secrets management

Traffic must be encrypted in transit. In production, TLS usually terminates at a load balancer or ingress (L04), but you should know Go serves it directly too:

srv.ListenAndServeTLS("cert.pem", "key.pem")
// or set srv.TLSConfig to pin a minimum version:
// tls.Config{MinVersion: tls.VersionTLS12}
Gotcha
Secrets (DB passwords, signing keys, API keys) come from the environment or a secrets manager (L04) — never committed to git or baked into an image. A leaked key in git history is the #1 security-review finding, and rotating it is painful. Treat .env as untracked.

Concept 6

Input validation & injection

Never trust input. The big families: SQL injection (fixed by parameterized queries, L05), command injection (avoid os/exec with user strings), and oversized payloads (a DoS).

r.Body = http.MaxBytesReader(w, r.Body, 1<<20)  // cap body at 1MB
var in CreateReq
dec := json.NewDecoder(r.Body)
dec.DisallowUnknownFields()                // reject junk fields
if err := dec.Decode(&in); err != nil {
    http.Error(w, "bad request", http.StatusBadRequest)  // 400
    return
}
// then validate fields: non-empty, in range, correct format
Idiom
Pair this with rate limiting (the L02 limiter, now as a defense) to blunt brute-force and abuse, and return generic error messages — don't leak "user not found" vs "wrong password," which tells an attacker which accounts exist.

Explain · recalls L03

Where in the request lifecycle do authentication and authorization run, and how does a handler learn who the user is?

In middleware (L03), before the handler. The authn middleware verifies the token/session, then stores the identity in the request context via context.WithValue and passes r.WithContext(ctx) down. The handler reads it back from r.Context(). Authorization can be a second middleware or an in-handler check against the user's roles.
Find the vulnerability

What's dangerously wrong here?

hash := sha256.Sum256([]byte(password))
// store hash; later:
if sha256.Sum256([]byte(attempt)) == hash { /* ok */ }
SHA-256 is a fast hash with no salt — an attacker with the DB can brute-force billions of guesses per second and use rainbow tables (identical passwords hash identically). Use bcrypt/argon2, which are deliberately slow and salt per-password. (Bonus: == on the hashes isn't constant-time.)
Explain the trade-off

Stateless JWTs vs server-side sessions — when would you pick each?

JWT: no per-request lookup, so it scales horizontally with no shared session store — great for many stateless services. Cost: you can't revoke a token before it expires, so keep them short and add refresh tokens. Sessions: instantly revocable and simple, but every request hits the session store (Redis) and you must share it across instances. Fintech often prefers sessions (or short JWTs + a revocation list) precisely because instant revocation matters.
Verify
On your P03 service: add the authn middleware, then curl a protected route with no token (expect 401), a tampered token (expect 401), and a valid one (expect 200). Then grep your repo for committed secrets: git log -p | grep -iE 'password|secret|key ='.
