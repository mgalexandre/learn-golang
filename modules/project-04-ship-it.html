Track B · Projects · P04 · production

Ship it: containers to cloud.

Take the P03 microservice from "runs on my machine" to "runs in production" — containerized, deployed, observable. Applies L04 · Containers and L06 · Observability.

🔨 Build module 📦 Applies L04 · L06 ⏱ ~1 week hands-on

What you'll ship

  • A ~15 MB multi-stage Docker image running as non-root
  • docker compose for local Postgres + Redis + service
  • Kubernetes manifests with liveness/readiness probes & resource limits
  • A GitHub Actions CI/CD pipeline: test → build → push
  • Structured logs, Prometheus /metrics, and OpenTelemetry traces
01

Dockerize (multi-stage)

Apply the L04 Dockerfile to the P03 service. Build static, ship on distroless.

FROM golang:1.25 AS build
WORKDIR /src
COPY go.* ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app ./cmd/server

FROM gcr.io/distroless/static
COPY --from=build /app /app
USER nonroot:nonroot
ENTRYPOINT ["/app"]
Verify
docker build -t weather . && docker images weather — confirm it's under ~20 MB. That number is the L04 lesson made concrete.
02

Compose the local stack

One command to bring up the whole system for development — the same Postgres + Redis P03/P05 assume.

services:
  db:    { image: postgres:16, environment: { POSTGRES_PASSWORD: dev } }
  redis: { image: redis:7 }
  api:
    build: .
    ports: ["8080:8080"]
    environment:
      DATABASE_URL: postgres://postgres:dev@db:5432/postgres
      REDIS_ADDR:   redis:6379
    depends_on: [db, redis]
Grill
Why does api read DATABASE_URL from the environment instead of a hardcoded string? (L04: config comes from the environment so the same image runs unchanged in compose, k8s, and AWS — only the injected values change.)
03

Kubernetes manifests

Wire the L04 probes to the P03 /healthz and /readyz endpoints. Set requests/limits and GOMEMLIMIT.

          livenessProbe:  { httpGet: { path: /healthz, port: 8080 } }
          readinessProbe: { httpGet: { path: /readyz,  port: 8080 } }
          env:
            - { name: GOMEMLIMIT, value: "230MiB" }   # below the limit
          resources:
            requests: { cpu: "100m", memory: "64Mi" }
            limits:   { cpu: "500m", memory: "256Mi" }
Verify
On a local cluster (kind or minikube): kubectl apply -f k8s/, then kubectl get pods — the pod should reach Running and 1/1 READY only after /readyz passes. Kill the DB and watch it go 0/1 without restarting.
04

CI/CD pipeline

Every push runs the L06 test suite with the race detector; main builds and pushes a SHA-tagged image.

jobs:
  ci:
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
      - run: go vet ./...
      - run: go test -race -cover ./...       # L02+L06 safety net
      - run: docker build -t $REG/weather:${{ github.sha }} .
      - if: github.ref == 'refs/heads/main'
        run: docker push $REG/weather:${{ github.sha }}
Grill
Why tag by git SHA rather than latest? (Immutable, traceable deploys and instant rollback — you always know exactly which commit is running. latest is ambiguous and un-rollback-able.)
05

Instrument it (the three pillars)

Now the L06 observability payoff. Add structured logs, a metrics endpoint, and traces so a slow request is diagnosable in prod.

// logs — slog JSON with a request id
logger.Info("served", "city", city, "ms", ms, "cache", hit)
// metrics — expose /metrics for Prometheus (RED: rate, errors, duration)
mux.Handle("/metrics", promhttp.Handler())
// traces — OpenTelemetry span per request, propagated via context
ctx, span := tracer.Start(ctx, "weather.Get"); defer span.End()
Verify
Hit the API under a small load (hey -n 200 localhost:8080/v1/weather/London), then curl localhost:8080/metrics and watch the request counter and latency histogram populate. That's the RED method live.
06

Deploy to the cloud

Push the image to a registry (ECR) and run it on managed infra — EKS (Kubernetes) or ECS (simpler). Use RDS for Postgres, ElastiCache for Redis, IAM roles for credentials (L04).

Grill
Final architecture defense: where do secrets come from in production, and why is an IAM role better than baking AWS keys into the image? (L04: roles issue short-lived, rotby credentials scoped least-privilege — nothing sensitive in the image or repo.)
