Testing & observability: proving it works, in prod.

"Reliability, observability, and security best practices" is a direct quote from the Lisbon listings. Go's testing is built in and excellent; observability is the three pillars — logs, metrics, traces.

Concept 1

Table-driven tests

Testing is in the standard library — no framework, files end in _test.go, run with go test ./.... The idiomatic style is a table of cases looped with subtests.

func TestCelsiusToF(t *testing.T) {
    tests := []struct{
        name string; in, want float64
    }{
        {"freezing", 0, 32},
        {"boiling",  100, 212},
        {"negative", -40, -40},
    }
    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {   // subtest per case
            if got := toF(tc.in); got != tc.want {
                t.Errorf("toF(%v) = %v, want %v", tc.in, got, tc.want)
            }
        })
    }
}
Idiom
t.Run gives each case its own name in the output and lets you run one with go test -run TestCelsiusToF/boiling. Add a case by adding a row — that's the whole appeal.

Concept 2 · why L01 interfaces mattered

Mocking through interfaces

Remember "accept interfaces" from L01? This is the payoff. If your service depends on an interface, tests can pass a fake — no network, no database, fast and deterministic.

type Fetcher interface {
    Fetch(ctx context.Context, city string) (Report, error)
}
// in the test: a fake that implements Fetcher
type fakeFetcher struct{ report Report; err error }
func (f fakeFetcher) Fetch(context.Context, string) (Report, error) {
    return f.report, f.err
}
Idiom
Because Go interfaces are satisfied implicitly, you don't need a mocking library — a struct with the right methods just works. Inject the fake to test error paths (timeout, 500) that are painful to trigger against a real dependency.

Concept 3

Testing HTTP handlers

net/http/httptest spins up handlers in-memory — no real port. This tests the L03 layer end to end, fast.

req := httptest.NewRequest("GET", "/weather/London", nil)
rec := httptest.NewRecorder()
handler.ServeHTTP(rec, req)

if rec.Code != http.StatusOK {
    t.Errorf("status = %d, want 200", rec.Code)
}

Concept 4

Benchmarks & the race detector

Go benchmarks live beside tests and measure allocations too. And -race is your L02 safety harness — run it in CI.

func BenchmarkDecode(b *testing.B) {
    for i := 0; i < b.N; i++ { decode(payload) }
}
// go test -bench . -benchmem    → ns/op and allocs/op
// go test -race ./...           → catches data races
Gotcha
Aim for meaningful coverage, not 100%. go test -cover reports it, but a test that asserts nothing inflates the number without protecting anything. Cover the branches that matter: error paths, edge cases, concurrency.

Concept 5 · pillar one

Structured logging with slog

Go 1.21+ ships log/slog — structured, leveled logging. In production you emit JSON so log aggregators (CloudWatch, Loki, Datadog) can index fields.

logger := slog.New(slog.NewJSONHandler(os.Stdout, nil))
logger.Info("weather served",
    "city", city,
    "duration_ms", ms,
    "cache", "hit")   // key/value pairs, not string soup
Idiom
Attach a request ID to a logger and pass it via context so every line for one request is correlated. Structured fields let you query "all requests where duration_ms > 500" — impossible with free-text logs.

Concept 6 · pillars two & three

Metrics & distributed tracing

Three pillars: logs (events), metrics (aggregatable numbers), traces (one request across services). Prometheus is the metrics standard; OpenTelemetry is the vendor-neutral tracing standard.

// Prometheus: a counter + a histogram
requests := prometheus.NewCounterVec(...)   // count by route+status
latency  := prometheus.NewHistogramVec(...) // p50/p95/p99
// expose /metrics; Prometheus scrapes it
The RED method
For every service, track Rate (requests/sec), Errors (failure %), and Duration (latency percentiles). Those three, plus a trace to drill into a slow request, are what "observability" means on a job spec — and what you'll wire up in Project P4.

Evidence-based practice

Retrieval drills

Predict before revealing. Drill 1 recalls L01 & L05.

Explain · recalls L01/L05

Your service struct holds a real Postgres *Repo. How do you unit-test its logic without a database?

Depend on an interface (e.g. ReadingStore), not the concrete *Repo. In production inject the Postgres implementation; in tests inject a fake struct that implements the same methods and returns canned data or errors. This is why "accept interfaces" (L01) is a testability decision, not a style one.
Find the weakness

A colleague reports "90% coverage" on this test. What's wrong?

func TestFetch(t *testing.T) {
    r, err := Fetch(ctx, "London")
    _ = r
    _ = err
}
It asserts nothing — it calls the code (so lines are "covered") but never checks the result or the error. Coverage measures lines executed, not behavior verified. It would pass even if Fetch returned garbage. Real tests assert on r and handle err with t.Fatal/t.Errorf.
Explain the trade-off

You have logs. Why also add metrics and traces — isn't that redundant?

They answer different questions. Metrics tell you that something's wrong cheaply and in aggregate ("error rate spiked to 5%") — you can't afford to log-scan for that at scale. Traces tell you where in a multi-service request the time or failure went. Logs give the detailed why for a specific event. Together they take you from "alert" → "which service" → "exact cause."
Verify
Run go test -race -cover ./... on your project. Then add a /metrics endpoint, hit your API a few times, and curl localhost:8080/metrics to watch the request counter tick up — observability you can see.
