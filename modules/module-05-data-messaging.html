Concept 1

database/sql, the standard interface

Go's database/sql is a driver-agnostic layer. For Postgres, the modern choice is pgx (used directly or via its stdlib adapter). Always use parameterized queries — the driver handles escaping.

row := db.QueryRowContext(ctx,
    "SELECT temp_c, description FROM readings WHERE city = $1", city)
var r Report
if err := row.Scan(&r.TempC, &r.Desc); err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return Report{}, ErrNotFound   // map to a domain error
    }
    return Report{}, err
}
Gotcha
Never build SQL with fmt.Sprintf and user input — that's SQL injection. Placeholders ($1 for Postgres, ? for MySQL) are non-negotiable. This is a guaranteed security-review flag.

Concept 2

Connection pools & context

An *sql.DB is not a connection — it's a pool, safe for concurrent use. Configure it, share one across your app, and always pass ctx so slow queries respect request timeouts (L02 + L03 come together here).

db.SetMaxOpenConns(25)                 // cap DB load
db.SetMaxIdleConns(25)
db.SetConnMaxLifetime(5 * time.Minute)  // recycle
Gotcha
Every Rows from QueryContext must be defer rows.Close()d, or you leak connections until the pool is exhausted and every request hangs. The single most common Go database bug in production.

Concept 3

Transactions done right

Fintech means money moving atomically. The Go pattern uses defer tx.Rollback() as a safety net — it's a no-op after a successful Commit.

tx, err := db.BeginTx(ctx, nil)
if err != nil { return err }
defer tx.Rollback()          // safety net if we return early

if _, err = tx.ExecContext(ctx, debit, from); err != nil {
    return err            // deferred Rollback fires
}
if _, err = tx.ExecContext(ctx, credit, to); err != nil {
    return err
}
return tx.Commit()        // Rollback after this is a no-op

Concept 4

Schema migrations

Schema changes are versioned files, applied in order, run in CI/CD before deploy. Tools: golang-migrate, goose, or atlas. Each migration has an up and a down.

-- 0002_add_readings.up.sql
CREATE TABLE readings (
  id      BIGSERIAL PRIMARY KEY,
  city    TEXT NOT NULL,
  temp_c  NUMERIC NOT NULL,
  taken_at TIMESTAMPTZ DEFAULT now()
);
Idiom
Migrations must be backward-compatible during a rolling deploy — old and new code run simultaneously. Add a column before you use it; drop it a release later. "Expand then contract."

Concept 5

Redis: the cache in front

Weather data is expensive to fetch and fine slightly stale — a perfect cache candidate. The cache-aside pattern: look in Redis, miss → hit the source → store with a TTL.

val, err := rdb.Get(ctx, "weather:"+city).Result()
if err == redis.Nil {           // cache miss
    report = fetchFromSource(ctx, city)
    rdb.Set(ctx, "weather:"+city, encode(report), 10*time.Minute)
}
Gotcha
Beware the thundering herd: when a hot key expires, thousands of requests miss at once and stampede the source. Mitigate with a short lock, request coalescing (singleflight), or jittered TTLs.

Concept 6 · the fintech one

Message queues: Kafka & RabbitMQ

Queues decouple services: a producer emits an event, consumers process it later, independently. Kafka = high-throughput event log (replayable, partitioned). RabbitMQ = flexible task/routing broker. Both appear across PT/LU listings.

// consumer loop — process, then acknowledge
for msg := range messages {
    if err := handle(ctx, msg); err != nil {
        // don't ack — it'll be redelivered
        continue
    }
    msg.Ack()   // only after success
}
At-least-once & idempotency
Most queues deliver at least once — a message can arrive twice (redelivery after a crash before ack). So consumers must be idempotent: processing the same event twice has the same effect as once (dedupe by event ID). This is the same idempotency idea from L03, now on the messaging side.

Explain · recalls L03/L04

Where should *sql.DB live in your service, and how does it reach a request handler?

Open it once at startup (it's a pool, concurrency-safe) and inject it — typically into a repository struct that handlers hold a reference to via dependency injection. Configure it from DATABASE_URL (L04 config). Never sql.Open per request: you'd spawn pools endlessly and exhaust the database.
Find the bug

This handler works in tests but exhausts connections in production. Why?

rows, _ := db.QueryContext(ctx, "SELECT city FROM readings")
for rows.Next() {
    var c string
    rows.Scan(&c)
    cities = append(cities, c)
}
return cities
rows is never closed. Each call holds its connection until GC eventually finalizes it — under load the pool drains and every request blocks waiting for a free connection. Fix: defer rows.Close() right after the query, and check rows.Err() after the loop. (Also: the query error is ignored.)
Explain the trade-off

Your consumer crashes after doing the work but before Ack(). What happens, and why must handlers be idempotent?

The broker never saw the ack, so it redelivers the message — the work runs a second time. With at-least-once delivery this is normal, not exceptional. If "the work" is "send a payment," a non-idempotent handler double-pays. Make it idempotent (e.g., record processed event IDs, or use upserts keyed by the event) so a duplicate is a safe no-op.
Verify
Spin up Postgres + Redis with docker compose up (ties to L04), point DATABASE_URL at it, insert a reading, and confirm a second identical request hits the cache — log a "HIT/MISS" line to prove it.
